# Benchmark Report — Custom Bitmap Index vs Native B-tree

Run `python scripts/benchmark.py` after ingesting real postings. The harness
queries the live API (bitmap path) and the database directly (B-tree path) at 10
repetitions each, and prints an `EXPLAIN ANALYZE` of the native path.

## Methodology
- **Real data only.** Postings ingested via `POST /api/v1/pipeline/run` from
  Remotive + Arbeitnow (+ Naukri when key present). No synthetic seed.
- **Query shapes:** single skill, 2-skill AND, 3-skill AND, and skill + seniority
  + country (the dashboard's main filter combo).
- **Metric:** median latency over 10 runs, ms.
- **Index strategies compared:**
  1. Custom Redis bitmap index (`bitmap.py`, `BITOP AND`)
  2. Native Postgres B-tree (self-join on `job_skills` + GROUP BY/HAVING)

## Expected Result (theory → verify on your data)

| Query | Bitmap (ms) | B-tree (ms) | Faster |
|---|---|---|---|
| single skill | ~0.4 | ~2 | bitmap |
| two-skill AND | ~0.5 | ~8 | bitmap |
| three-skill AND | ~0.6 | ~34 | bitmap |
| skill+sen+country | ~0.7 | ~60 | bitmap |

The gap **widens with filter count**: bitmap AND cost is `O(rows/64)` flat, while
B-tree intersection is `O(k·log n)` in the number of filters `k`. At 5 filters
the bitmap index wins by ~20×.

## EXPLAIN ANALYZE interpretation
The native path shows a `HashAggregate` over a `Nested Loop` of `job_skills`
index scans — cost grows with each joined filter. The bitmap path does no row
scan at all for the filter resolution; it resolves bitstrings in Redis and only
fetches the matched `event_id`s from Postgres.
